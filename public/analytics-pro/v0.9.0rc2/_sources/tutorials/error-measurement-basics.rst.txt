In previous tutorials, we saw how to write differentially private queries using the Tumult Analytics API. Differential privacy adds noise to data, and also involves truncation, so before going ahead with publishing or sharing DP statistics, we typically want to evaluate how accurate these statistics are, and carefully select the parameters used in our Tumult Analytics program.

That's exactly what this tutorial series about error measurement and tuning is about. In this tutorial we will walk you through the first step of the process of measuring and optimizing error: converting a simple program to have a fixed structure, and evaluate its error. Let's get started!

Setup
-----

First, let's import the Python packages we will use, and download some sample data.

Throughout this tutorial section, we will use two tables: `members` contains information about the members of our fictional public library, and `nc-zip-codes` lists all the ZIP codes in North Carolina, where our library is located.

.. testcode::

    import matplotlib.pyplot as plt

A simple query
--------------

Say we want to compute the number of books borrowed by users of our library, sliced by ZIP code.  We know how to do this, thanks to what we learned in previous tutorials. Below, we use two input Spark DataFrames: `members_df`, which contains a list of members of our fictional public library and information such as how many books each one borrowed, and `nc_zip_codes_df`, which contains a list of ZIP codes in North Carolina. The former needs to be protected, the second is public.

.. testcode::

    session = Session.from_dataframe(
        source_id="members",
        dataframe=members_df,
        protected_change=AddOneRow(),
        privacy_budget=PureDPBudget(epsilon=3),
    )

    zip_code_keys = KeySet.from_dataframe(nc_zip_codes_df.select("zip_code"))
    query = (
        QueryBuilder("members")
        .groupby(zip_code_keys)
        .sum("books_borrowed", low=0, high=500)
    )
    books_by_zip_code = session.evaluate(query, session.remaining_privacy_budget)

If we run this program, we get a DataFrame with noisy sums and we can use a visualization library to plot this into a nice map. Let's consider only top ten ZIP codes in which most books have been borrowed and display the noisy sums as a graph.

.. code-block::
    
    import matplotlib.pyplot as plt
    import seaborn as sns

    sns.set_theme(style="whitegrid")
    top_10_zipcodes = books_by_zip_code.orderBy(col("books_borrowed_sum").desc()).limit(10)
    data_to_plot = top_10_zipcodes.toPandas()

    g = sns.barplot(x="zip_code", y="books_borrowed_sum", data=data_to_plot, color="#1f77b4")
    g.set_xticklabels(
        data_to_plot["zip_code"], rotation=45, horizontalalignment="right"
    )
    plt.title("The ten ZIP codes in which most books have been borrowed")
    plt.xlabel("ZIP code")
    plt.ylabel("Total books borrowed")
    plt.tight_layout()
    plt.show()

However, it's hard to know how trustworthy these results are. Let's figure it out by computing the error of the above query. To get a sense of the "typical" error in these noisy statistics, we will compute the relative error for each zip code and then report the median value of those errors.

From Session to SessionProgram
------------------------------

To get an error report, we first need to convert our program to a :class:`~.SessionProgram` instead. It works similarly to a Session, but the operations are defined in the reverse order: first we will describe all the queries we want to run on the data, and then we will initialize a :class:`~.SessionProgram` with the data and the privacy guarantees we want to enforce.

.. testcode::

    class BooksByZipCodeProgram(SessionProgram):
        class ProtectedInputs:
            members: DataFrame
        class UnprotectedInputs:
            nc_zip_codes: DataFrame
        class Outputs:
            books_by_zip_code: DataFrame
        
        def session_interaction(self, session):
            nc_zip_codes_df = session.public_source_dataframes["nc_zip_codes"]
            nc_zip_code_keys = KeySet.from_dataframe(nc_zip_codes_df.select("zip_code"))
            query = (
                QueryBuilder("members")
                .groupby(nc_zip_code_keys)
                .sum("books_borrowed", low=0, high=500)
            )
            budget = session.remaining_privacy_budget
            return {"books_by_zip_code": session.evaluate(query, budget)}

Let's look at each part separately.

1. First, we declare our program as a Python class subclassing :class:`~.SessionProgram`. We indicate what are its inputs (the protected ones separately from the non-protected ones) and its outputs.
2. Second, we write our program logic inside a :meth:`~.session_interaction` method. This method takes a Session as argument, and uses it to evaluate queries as we previously did. It then returns a dictionary mapping each output name with the corresponding output DataFrame.

Initializing and running a SessionProgram
-----------------------------------------

The only way that a SessionProgram can process its input data is by interacting with a :class:`Session<tmlt.analytics.session.Session>` in the :meth:`~.session_interaction` method. Its privacy guarantee is exactly the same as a :class:`Session<tmlt.analytics.session.Session>`'s: the data is protected with fixed privacy guarantees upon initialization, using the same syntax as the Session; once initialized, it will enforce these privacy guarantees in all future interactions with it. Let's initialize our `BooksByZipCodeProgram`.

.. testcode::

    zip_code_program = (
        BooksByZipCodeProgram.Builder()
        .with_private_dataframe(
            source_id="members",
            dataframe=members_df,
            protected_change=AddOneRow()
        )
        .with_public_dataframe(
            source_id="nc_zip_codes",
            dataframe=nc_zip_codes_df,
        )
        .with_privacy_budget(PureDPBudget(3))
        .build()
    )

Once initialized, instead of calling queries on it like the Session, we run the whole program at once using the :meth:`~.run` method, and obtain the outputs with `outputs`.

.. testcode::

    zip_code_program.run()
    zip_code_program.outputs["books_by_zip_code"].show(10)

One characteristic of a :class:`~.SessionProgram` is that it can only be run *once*: running it again is forbidden.

.. doctest::

    >>> zip_code_program.run()
    Traceback (most recent call last):
    ...
    RuntimeError: The SessionProgram has already been run. It can only be run once.

First look at error
-------------------

Now that we have clearly defined our program, and specified its inputs, outputs, and query logic, we can pass this information to the :class:`SessionProgramTuner<tmlt.analytics.tuner._tuner.SessionProgramTuner>` class, which is what gives us access to error reports.

Defining this Tuner works by subclassing the :class:`SessionProgramTuner<tmlt.analytics.tuner._tuner.SessionProgramTuner>` class, passing our program as a class argument. Tumult Analytics provides built-in metrics that we can use to measure the error of our program. To calculate median relative error, we instantiate the built-in metric :class:`MedianRelativeError<tmlt.analytics.metrics._relative_error.MedianRelativeError>` and include it in the list of metrics associated with the ``metrics`` class variable. This is what happens behind the scenes: one baseline table, one DP table; both are joined, then the numbers are compared. We need to specify on what column names we must do the join, and what column contains the numbers.

.. testcode::

    class SimpleTuner(SessionProgramTuner, program=BooksByZipCodeProgram):
        metrics = [
            MedianRelativeError(output="books_by_zip_code", measure_column="books_borrowed_sum", join_columns=["zip_code"]),
        ]

The :class:`SessionProgramTuner<tmlt.analytics.tuner._tuner.SessionProgramTuner>` can be initialized just like a Session, but **it does not provide any privacy guarantees**: error information is not differentially private! For this reason, it is a good practice to use different datasets for tuning and deployment — historical datasets or synthetic datasets are common choices.

.. testcode::

    simple_tuner = (
        SimpleTuner.Builder()
        .with_private_dataframe(
            source_id="members",
            dataframe=members_df,
            protected_change=AddOneRow()
        )
        .with_public_dataframe(
            source_id="nc_zip_codes",
            dataframe=nc_zip_codes_df,
        )
        .with_privacy_budget(PureDPBudget(3))
        .build()
    )

Now that our :class:`SessionProgramTuner<tmlt.analytics.tuner._tuner.SessionProgramTuner>` is initialized, we can get our very first error report by calling the :meth:`~tmlt.analytics.tuner._tuner.SessionProgramTuner.error_report` method.

.. testcode::

   error_report = simple_tuner.error_report()
   error_report.show()
